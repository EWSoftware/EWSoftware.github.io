    Additional Notices Plug-In

    Notices appear at the top of topics and as tags in the member list pages to show additional important information about API members. By default, notices are added for any type or member with a preliminary XML comments element and those decorated with the System.ObsoleteAttribute or System.Diagnostics.CodeAnalysis.ExperimentalAttribute attribute. This plug-in is used to add additional notice definitions to the selected presentation style based on your own custom attributes or XML comments elements.

    Configuration Options

    The configuration options consist of the following properties:

    • Attribute Type Name - If the notice is related to an attribute on a type or member, specify the fully qualified name here including the "T:" prefix. Example: T:System.ObsoleteAttribute.

    • XML Comments Element Name - If the notice is related to an XML comments element on a type or member, specify the element name here. Example: preliminary.

    • Use attribute/element value for notice message text - If this option is checked, the inner text of the XML comments element or the attribute value will be used as the notice message. If not checked, the notice message from the definition will be used. If the comments element is empty or the attribute has no value, the notice text in the definition is used automatically.

    • Notice Message - This is the message to display at the top of the API member topics. It can be literal text or a localized content item ID prefixed with '@'. For example, @boilerplate_obsoleteLong is used for the obsolete notice message. If a notice message is not specified, a notice message will not appear for the notice in member topics.

    • Tag Text - This is the tag text to display below the member descriptions on member list pages. It can be literal text or a localized content item ID prefixed with '@'. For example, @boilerplate_obsoleteShort is used for the obsolete notice tag. Tags are typically one or two words. Reserve the longer description for the notice message shown at the top of the topic. If tag text is not specified, a tag will not appear for the notice.

    • Notice Style Classes - For HTML presentation styles, this allows you to set the CSS classes that will be applied to the notice message. If not specified, default styles will be applied. For example, in the Default 2022 presentation style the default styles are tag is-info is-medium. You can use various combinations of styles on the notice message and tag. Some examples are shown below:

      tag is-primary is-lighttag is-link is-lighttag is-info is-lighttag is-success is-lighttag is-warning is-lighttag is-danger is-light
      tag is-link is-normaltag is-primary is-mediumtag is-info is-largetag is-success is-normal is-rounded

    • Tag Style Classes - For HTML presentation styles, this allows you to set the CSS classes that will be applied to the tags on member list pages. If not specified, default styles will be applied. For example, in the Default 2022 presentation style the default styles are tag is-info is-rounded. See above for some example styles.

    Localizing Notice Message and Tag Text

    As noted above, if the notice message or tag text specify a content item ID prefix with '@', the related text for the item in the resource item file will be used. You can specify localized text for your own notice definitions by adding one or more resource items files to the project. The filenames must have a suffix identifying the language of the items they contain prefixed with an underscore and have a .items filename extension. Examples: ProjectNotices_en-US.items, ProjectNotices_fr-Fr.items. The file's Build Action should be set to ResourceItems.

    The content of the notice content items file will look like the following example:

     
    <?xml version="1.0" encoding="utf-8"?>
<content xml:space="preserve">
  <item id="unsafeNoticeMessage">This API does some unsafe stuff</item>
  <item id="unsafeTag">Unsafe</item>
  <item id="topicNoticeMessage">This is a notice for a custom XML comments element</item>
  <item id="topicNoticeTag">Notice</item>

  <!-- If you want to override the preliminary, obsolete, or experimental notice text,
     you can add the following entries to your file with your own message and tag text.
  <item id="boilerplate_obsoleteShort">Obsolete</item>
  <item id="boilerplate_obsoleteLong">Note: This API is now obsolete.</item>
  <item id="boilerplate_experimentalShort">Experimental</item>
  <item id="boilerplate_experimentalLong">Note: This API is for evaluation purposes only and is subject to change or removal in future updates.</item>
  <item id="preliminaryApi">This API is preliminary and subject to change.</item>
  <item id="preliminaryShort">Preliminary</item>
  -->
</content>

      Note

    Resource item files have a custom editor and are usually used to override default help file builder content items. To add the initial message items for your notices, open the file with a text editor. Once the items have been added, you can open it with the custom editor to make changes to the new items if wanted.

    Overriding the Preliminary, Obsolete, and Experimental Notices

    As noted, the preliminary XML comments element and the obsolete and experimental attributes are added by default. If you want to override the messages or styles used, you can add matching notice entries to the plug-in configuration for the element or attribute name. If you just want to change the notice message or tag text, you can also do so through the resource items file as shown in the example above without using the plug-in.
